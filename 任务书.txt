AI 智能招聘系统 · 项目任务书 v1.0
一、项目定位
项目名称	AI 智能招聘系统（AI Recruitment System）
核心目标	用 AI 替代人工完成简历初筛，通过大模型 API 对简历与岗位进行量化匹配打分，帮助 HR 从"看简历"变成"看分数+理由"
核心价值	降低简历筛选人力成本，提升岗位匹配精准度，让招聘决策有据可查
目标用户	企业内部 HR、招聘负责人、用人部门 Leader
二、角色分析（12 角色视角）

按照文档里的多角色拆解方法，从 12 个视角审视这个系统：

角色	关注点	对系统的核心诉求
HR 专员	日常操作效率	一键上传简历，直接看分数和推荐结论
招聘负责人	整体进度把控	看到每个岗位的候选人漏斗和评分分布
用人部门 Leader	专业匹配度	评分维度必须覆盖技术技能，不能只看学历
候选人	公平透明	评分逻辑要可解释，不是黑盒
系统管理员	权限和稳定性	用户角色隔离，API 调用不能失控
数据安全负责人	合规	简历是个人隐私数据，存储和传输要合规
财务	成本控制	大模型 API 调用成本要可预测、可限额
开发工程师	可维护性	模块解耦，打分逻辑可以独立替换
测试工程师	质量保证	打分结果要稳定，同一份简历多次打分不能差异过大
运维工程师	部署和监控	服务要可观测，API 超时要有降级
业务分析师	数据价值	沉淀历史打分数据，分析招聘漏斗效率
法务	合规风险	AI 打分不能成为歧视性决策的唯一依据

四、各模块详细定义
模块 A：用户与权限管理

目标：不同角色看到不同功能，互不干扰。

角色	权限范围
超级管理员	所有权限 + 系统配置 + API Key 管理
HR 专员	发布岗位、上传简历、查看打分、推进候选人状态
部门 Leader	只读查看本部门岗位的候选人和评分
数据分析员	只读查看所有数据看板

功能点：

登录 / 登出（JWT Token 认证）

用户 CRUD

角色分配

操作日志（谁在什么时候做了什么）

模块 B：岗位管理

目标：JD（岗位描述）是打分的基准，必须结构化存储。

功能点：

创建 / 编辑 / 关闭岗位

岗位字段：职位名称、所属部门、工作地点、薪资范围、学历要求、工作年限要求、技能要求（标签）、JD 正文

岗位状态：草稿 / 招聘中 / 已暂停 / 已关闭

支持从模板创建岗位

岗位列表搜索和筛选

关键约束：

JD 正文是大模型打分的核心输入，创建岗位时必填，字数不少于 100 字

岗位关闭后不允许新增候选人

模块 C：简历管理

目标：把简历从非结构化文件变成系统内可处理的数据。

功能点：

支持上传 PDF、DOCX 格式简历

自动提取简历文本（pdfplumber / python-docx）

简历基础信息解析（姓名、联系方式、工作年限、学历）

简历关联到具体岗位

简历状态：待评分 / 评分中 / 已评分 / 评分失败

支持批量上传（一次上传多份简历关联同一岗位）

原始文件安全存储（MinIO / OSS）

关键约束：

文件大小限制：单份不超过 10MB

简历文本提取失败时，系统明确提示原因，不静默失败

简历原文件只有 HR 和管理员可以下载

模块 D：AI 简历匹配打分（核心模块）

目标：调用大模型 API，对简历和 JD 进行量化打分，输出结构化评分报告。

D1. 打分触发方式
方式	场景
上传时自动触发	简历上传成功后自动进入打分队列
手动触发单份	HR 对某份简历手动重新打分
批量触发	对某岗位所有未评分简历批量触发
D2. 打分维度
维度	权重	评分说明
技能匹配	25 分	技术栈、工具、证书与 JD 要求的重合度
经验匹配	25 分	工作年限、行业背景、项目经历相关性
教育背景	20 分	学历层次、专业相关性
发展潜力	15 分	成长轨迹、项目复杂度、自我驱动迹象
稳定性	15 分	平均在职时长、跳槽频率
总分	100 分	
D3. 打分输出结构
{
  "total_score": 82,
  "dimensions": {
    "skills_match":      { "score": 22, "max": 25, "comment": "..." },
    "experience_match":  { "score": 20, "max": 25, "comment": "..." },
    "education_match":   { "score": 16, "max": 20, "comment": "..." },
    "potential":         { "score": 13, "max": 15, "comment": "..." },
    "stability":         { "score": 11, "max": 15, "comment": "..." }
  },
  "strengths":    ["...", "...", "..."],
  "weaknesses":   ["...", "..."],
  "recommendation": "Recommended",
  "summary": "候选人在 XX 方面表现突出，但 XX 方面存在差距..."
}
D4. 打分稳定性保证

大模型 temperature 设置为 0.2，减少随机性

强制 JSON 输出（response_format: json_object）

同一份简历如重复打分，分数波动超过 10 分时系统告警

打分结果连同原始 prompt 一起存储，便于审计和复现

D5. API 成本控制

每份简历打分前估算 token 数，超过阈值（如 8000 tokens）先做文本截断

设置月度 API 调用预算上限，超限后停止自动打分并告警

打分结果缓存，相同简历 + 相同 JD 不重复调用

D6. 降级策略
场景	处理方式
API 超时（> 30s）	重试 2 次，仍失败标记为"评分失败"，不阻塞流程
API 返回非 JSON	重试 1 次，失败后人工介入
API 额度耗尽	停止自动打分，HR 收到通知，已有分数不受影响
简历文本为空	跳过打分，标记"无法提取文本"，提示人工处理
模块 E：候选人管理与流程推进

目标：打分只是第一步，后续还要有人工决策流程。

候选人状态机：

待评分 → 已评分 → HR 初筛 → 部门审核 → 面试安排 → Offer → 入职 / 淘汰
                                                         ↑
                                              每个节点都可以标记淘汰

功能点：

按岗位查看候选人列表，默认按 AI 评分从高到低排序

支持按分数段筛选（如只看 70 分以上）

HR 可以在 AI 评分基础上添加人工备注

状态流转记录（谁在什么时候把候选人推进到哪个状态）

面试安排（时间、面试官、形式）

候选人沟通记录（电话/邮件记录）

关键约束：

AI 评分只作为参考，最终决策必须由人工确认

界面上明确标注"AI 辅助评分，仅供参考"

任何状态变更必须有操作人和时间戳

模块 F：数据看板与分析

目标：让招聘负责人看清整体漏斗，让 AI 打分的价值可量化。

看板内容：

指标	说明
各岗位候选人数量	实时统计
评分分布直方图	每个岗位的评分分布情况
招聘漏斗	各阶段转化率
AI 推荐 vs 人工结果对比	AI 推荐通过的人最终面试通过率，用于验证打分有效性
平均评分时长	API 响应效率监控
API 调用量和费用趋势	成本可视化
模块 G：系统配置与 API 管理

功能点：

大模型 API Key 配置（支持 OpenAI / 通义千问 / 智谱 GLM 切换）

打分 Prompt 模板管理（HR 可以调整打分侧重点）

API 月度预算设置

系统通知配置（评分失败、额度告警等）

数据导出（候选人列表 + 评分结果导出为 Excel）

七、逻辑自洽性检查（红队视角）

在动代码前，先在逻辑层找漏洞：

风险点	问题描述	解决方案
打分歧视风险	大模型可能对年龄、性别等信息打分有偏见	Prompt 明确禁止考虑年龄、性别、籍贯等非工作相关因素
打分不稳定	同一份简历多次打分结果差异大	temperature 0.2 + 分数波动告警
API 成本失控	批量打分时突发大量调用	异步队列 + 月度预算硬限制
简历隐私泄露	简历文本直接发给第三方 API	数据处理协议确认 + 脱敏处理 + 存储加密
HR 过度依赖 AI	AI 分低的候选人直接被淘汰不看	流程上要求 HR 对分数段内手动抽检
打分黑盒	候选人或部门质疑打分结果无从解释	每份评分报告保留打分维度明细和文字说明
JD 质量差导致打分失准	JD 写得太模糊，大模型无从判断	创建岗位时对 JD 长度和关键字段做验证
八、开发阶段划分
阶段	内容	预估周期
Phase 1	用户权限 + 岗位管理 + 简历上传解析	2 周
Phase 2	AI 打分核心模块 + 异步队列	2 周
Phase 3	候选人流程管理 + 状态机	1 周
Phase 4	数据看板 + API 管理配置	1 周
Phase 5	测试、安全加固、部署	1 周
九、验收标准

上传一份 PDF 简历，系统能在 60 秒内返回结构化评分报告

同一份简历对同一 JD 重复打分 5 次，分数波动不超过 5 分

API 调用失败时，系统不崩溃，候选人状态正确标记为"评分失败"

批量上传 20 份简历，全部完成打分，无遗漏

所有写操作有操作日志可查

HR 和部门 Leader 账号权限严格隔离，互不越权

十、一句话定位

这不是一个"有 AI 功能的招聘系统"，而是一个以 AI 打分为核心决策辅助手段的招聘管理平台，HR 的工作从"读简历"变成"审核 AI 的判断"。
